## 1. ما هو super\_admin؟

`super_admin` هو أعلى مستوى صلاحية في نظام Resort OS (المستوى 100، فوق
`admin` المستوى 80). يقع خارج التسلسل الهرمي العادي للأدوار ولا يمكن تجاوز
صلاحياته بأي منع صريح.

**يجب أن يكون هناك دائمًا حسابان نشطان على الأقل بصلاحية super\_admin.**
إذا كان عندك حساب واحد فقط وتعطّل أو تعرّض للاختراق، فإن الاسترداد يتطلب
وصولًا مباشرًا للسيرفر. يطبّق النظام هذا القيد تلقائيًا — سيرفض تعطيل
أو تخفيض آخر super\_admin نشط.

`super_admin` **حساب شخصي مُسمّى، لا يُشارَك أبدًا.** لكل مشغّل بياناته
الخاصة وجهاز TOTP الخاص ورموز الاسترداد الخاصة.

## 2. تسجيل الدخول والمصادقة الثنائية الإجبارية (TOTP)

### الإعداد الأول

1. استلم بيانات الاعتماد المؤقتة من مدير النظام (البريد الإلكتروني +
   كلمة المرور المؤقتة + رمز التسجيل).
2. انتقل إلى `https://[نطاق-المنتجع]/login` وسجّل الدخول.
3. سيُجبَر على تغيير كلمة المرور فورًا (12 حرفًا كحد أدنى).
4. بعد تغيير كلمة المرور ستُحال إلى `/2fa-setup`.
5. امسح رمز QR بتطبيق مصادقة (Google Authenticator, Aegis, Authy، أو أي
   تطبيق TOTP متوافق).
6. **دوّن رموز الاسترداد واحتفظ بها خارج الجهاز** — طباعتها ووضعها في
   خزنة مادية هو الأسلوب الموصى به. تُعرض مرة واحدة فقط.
7. أدخل الرمز المكوّن من 6 أرقام من التطبيق لتأكيد التسجيل.

### كل تسجيل دخول لاحق

1. أدخل البريد الإلكتروني وكلمة المرور.
2. أدخل رمز TOTP الحالي المكوّن من 6 أرقام.
3. يرفض النظام تسجيل الدخول بدون TOTP صالح — لا يوجد استثناء في الإنتاج.

### فقدان جهاز المصادقة — الاسترداد

استخدم أحد رموز الاسترداد المحفوظة بدلًا من رمز TOTP في شاشة تسجيل الدخول.
كل رمز استرداد يُستخدم مرة واحدة فقط. بعد استعادة الوصول، انتقل فورًا إلى
**الحساب ← الأمان** لتوليد رموز استرداد جديدة وإعادة تسجيل TOTP.

إذا استنفدت جميع رموز الاسترداد، راجع [إجراءات الطوارئ](#11-إجراءات-الطوارئ).

## 3. ما يستطيع وما لا يستطيع super\_admin فعله

### ✅ مسموح به

| المجال | الإجراءات |
|---|---|
| المستخدمون | إنشاء حسابات الموظفين، تغيير الأدوار، تفعيل/تعطيل الحسابات |
| الصلاحيات | منح أو منع صلاحيات محددة لأي مستخدم (غير super_admin) |
| الإعدادات العامة | قراءة وتحديث إعدادات النظام بأكمله |
| إعدادات الفروع | قراءة إعدادات الفروع؛ تحديثها بالنطاق الصحيح |
| سجل التدقيق | قراءة جميع الأحداث (للقراءة فقط) |
| الجلسات | عرض جلساتك الخاصة وإلغاء أي جلسة |
| الأمان | عرض نشاط الأمان، إدارة المصادقة الثنائية الخاصة بك |

### ❌ غير مسموح به (بتصميم مقصود)

| الإجراء | السبب |
|---|---|
| منح أو منع صلاحيات لحساب super\_admin آخر | المبدأ #2 — صلاحية super_admin لا يمكن تقليصها بأي منع صريح |
| تخفيض دور أو تعطيل آخر super\_admin نشط | المبدأ #4 — يجب وجود super_admin نشط دائمًا |
| تخفيض دور أو تعطيل حسابك الخاص من لوحة التحكم | المبدأ #3 — منع الإقفال العرضي للنفس |
| تعديل طلبات أو مدفوعات أو قيود محاسبية مُسجَّلة | السجلات المالية مُقفلة بعد الترحيل |
| حذف أو تعديل سجل التدقيق | سجل التدقيق للإلحاق فقط، لا يُحذف |
| الاطلاع على كلمات المرور أو الرموز أو مفاتيح التشفير | لا تُعرض في الواجهة أبدًا |
| تشغيل أوامر SQL أو Shell من الواجهة | غير متاح |
| الكتابة إلى إعدادات غير موجودة في السجل المعتمد | يرفضها المُدقِّق من جانب السيرفر |

## 4. مصادقة Step-Up — كيف تعمل

كل عملية حساسة — إنشاء حسابات، تغيير أدوار، منح أو منع صلاحيات، تعديل
إعدادات عامة — تتطلب **مصادقة Step-Up** إضافية فوق جلستك النشطة.

### كيف تتم العملية

1. تبدأ إجراءً حساسًا (مثلًا: تغيير دور موظف).
2. يظهر مربع حوار للتأكيد يطلب منك:
   - **كلمة مرورك الحالية.**
   - **رمز TOTP الحالي** (إلزامي لـ super\_admin).
   - **سبب** — نص حر، 3 أحرف كحد أدنى، يُسجَّل في سجل التدقيق.
3. يُصدر النظام رمز Step-Up أحادي الاستخدام صالحًا لمدة ~5 دقائق. الرمز
   مرتبط تشفيريًا بالعملية المحددة التي أعلنتها فقط — لا يمكن إعادة
   استخدامه لعملية مختلفة.
4. يُستهلك الرمز ذريًا في لحظة تنفيذ التعديل.

### لماذا السبب إلزامي؟

السبب يُسجَّل مع المنفّذ والهدف والوقت وعنوان IP ومعرف الطلب. كل عملية
حساسة تُنتج حدثًا قابلًا للإسناد بلا استثناء.

### انتهاء صلاحية الرمز

إذا مضت ~5 دقائق بين تأكيد Step-Up والضغط على "تأكيد"، يصبح الرمز منتهيًا.
ترفض العملية بـ `STEP_UP_REQUIRED`. أعد تأكيد هويتك للمتابعة.

## 5. إدارة المستخدمين

انتقل إلى **الإدارة ← 🛡️ لوحة تحكم Super Admin ← تبويب المستخدمون**، أو
**الإدارة ← 👤 حسابات الموظفين**.

### إنشاء حساب موظف جديد

1. أدخل: الاسم الكامل، البريد الإلكتروني، الدور، اللغة المفضّلة. الهاتف
   وربط سجل الموارد البشرية اختياريان.
2. انقر **إنشاء الحساب**.
3. أكمل Step-Up (كلمة المرور + TOTP + السبب).
4. يظهر لوحة بيانات الاعتماد تحتوي على:
   - **كلمة المرور المؤقتة** — يجب على الموظف تغييرها عند أول تسجيل دخول.
   - **رمز التسجيل (Enrollment Token)** — مطلوب لإتمام إعداد TOTP عند أول
     تسجيل دخول.
5. **انسخ هذه البيانات وأرسلها بأمان للموظف. تُعرض مرة واحدة فقط.** بعد
   إغلاق اللوحة لا يمكن استرداد بيانات الاعتماد من الواجهة.

### الأدوار المتاحة (من الأعلى للأدنى)

| الدور | المستوى | الاستخدام المعتاد |
|---|---|---|
| `super_admin` | 100 | مالك المنتجع / مدير تقنية المعلومات — يُسنَد عبر bootstrap فقط |
| `admin` | 80 | مدير المنتجع، وصول كامل للمكتب الخلفي |
| `accountant` | 70 | فريق المالية — 2FA إلزامي |
| `hr_manager` | 70 | وحدة الموارد البشرية |
| `manager` | 60 | مدير فرع / منفذ |
| `supervisor` | 50 | مشرف الدور |
| `receptionist` | 40 | موظف الاستقبال |
| `cashier` | 40 | نقطة البيع / كاشير الوردية |
| `waiter` | 30 | الخدمة، إدخال الطلبات |
| `chef` / `kitchen` | 30 | شاشة المطبخ |
| `employee` | 20 | بوابة الخدمة الذاتية فقط |

### تغيير الدور

1. ابحث عن المستخدم، انقر **تعديل**.
2. اختر الدور الجديد وأكّد.
3. أكمل Step-Up. تُلغى جميع الجلسات النشطة لذلك المستخدم فورًا — يجب عليه
   تسجيل الدخول مجددًا.

### تفعيل / تعطيل الحساب

انقر **تفعيل** أو **تعطيل**. يتطلب Step-Up. تعطيل المستخدم يُلغي جميع
جلساته ورموز الوصول الخاصة به فورًا.

## 6. إدارة الصلاحيات

انتقل إلى **الإدارة ← 🛡️ لوحة تحكم Super Admin ← تبويب الصلاحيات**، أو
**الإدارة ← 🔐 الصلاحيات**.

### الحالات الثلاث

| الحالة | المعنى |
|---|---|
| **افتراضي** | الوصول يتبع مستوى دور المستخدم |
| **ممنوح** | مسموح صراحةً، حتى لو لم يكن الدور يُتيحه عادةً |
| **محجوب** | محظور صراحةً، حتى لو كان الدور يُتيحه عادةً |

### القواعد

- لا يمكن تعيين أي استثناء لحساب `super_admin` — يرفضه السيرفر.
- إزالة الاستثناء (العودة للافتراضي) تتطلب Step-Up بسبب أيضًا.
- الاستثناءات على مستوى النظام بالكامل افتراضيًا ما لم تحدد نطاق فرع.
- يمكنك فقط منح/حجب صلاحيات موجودة في كتالوج السيرفر المعتمد.

### سير العمل

1. اختر موظفًا من القائمة على اليسار.
2. تُعرض جميع الصلاحيات على اليمين مُجمَّعة حسب الوحدة، مع تمييز الحالة
   الحالية.
3. انقر **منح** أو **افتراضي** أو **حجب** في أي صف.
4. أكمل Step-Up.

## 7. الإعدادات العامة

انتقل إلى **الإدارة ← 🛡️ لوحة تحكم Super Admin ← تبويب الإعدادات**.

> ⚠️ **الإعدادات العامة تؤثر على النظام بأكمله.** غيّرها فقط حين تفهم
> التأثير. بعضها يسري فورًا والبعض يتطلب إعادة تشغيل الخدمة.

### تعديل إعداد

1. انقر **تعديل** في الصف المطلوب.
2. أدخل القيمة الجديدة.
3. أكمل Step-Up (كلمة المرور + TOTP + السبب).

### ما لا يمكن تغييره من هنا

- المفاتيح غير الموجودة في السجل المعتمد يرفضها السيرفر.
- العلامات للقراءة فقط (إصدار قاعدة البيانات، حالة الترحيل) لا يمكن
  تعديلها.
- القيم السرية (مفاتيح API، مفاتيح التشفير، بيانات خدمات خارجية) يُديرها
  مدير النظام عبر متغيرات البيئة على السيرفر — لا تُعرض في الواجهة أبدًا.

## 8. سجل التدقيق

انتقل إلى **الإدارة ← 🛡️ لوحة تحكم Super Admin ← تبويب التدقيق**.

سجل التدقيق **للقراءة فقط وللإلحاق فقط**. كل إجراء حساس يُسجَّل تلقائيًا.
لا يمكن حذف أي إدخال أو تعديله أو إخفاؤه.

كل إدخال يحتوي على:

| الحقل | الوصف |
|---|---|
| الإجراء | ما حدث (`user.role_changed`, `permission.granted`, إلخ) |
| الكيان | ما تغيّر (النوع + المعرّف) |
| المنفّذ | من نفّذ الإجراء (معرّف المستخدم) |
| الوقت | التوقيت بالتنسيق UTC |
| التفاصيل | السبب، مرجع Step-Up، أسلوب التحقق، عنوان IP |

استخدم فلترات **الإجراء** و**الكيان** لتضييق النتائج. 50 إدخالًا لكل صفحة.

## 9. إدارة الجلسات

انتقل إلى **الحساب ← الجلسات** (`/account/sessions`).

تستطيع رؤية جميع جلساتك النشطة (كل جهاز أو متصفح مسجّل الدخول)، بما فيها
تاريخ البدء التقريبي وآخر نشاط والجهاز وعنوان IP.

- **إلغاء جلسة واحدة:** انقر إلغاء بجانبها. يتطلب Step-Up.
- **إلغاء جميع الجلسات الأخرى:** يُنهي كل الجلسات ما عدا الجلسة الحالية.
  يتطلب Step-Up.

إلغاء الجلسات لا يؤثر على سجل التدقيق.

## 10. قواعد الأمان الواجبة

1. **لا تشارك بيانات اعتمادك أو جهاز TOTP مطلقًا.** كل حساب يجب أن يبقى
   حصرًا شخصيًا.
2. **احتفظ برموز الاسترداد خارج الجهاز** — طباعة ووضع في خزنة مادية، ليس
   ملفًا على نفس الحاسوب.
3. **اكتب دائمًا سببًا ذا معنى** عند الطلب. الإدخالات المبهمة ("تجربة"،
   "موافق") تُفقد سجل التدقيق قيمته عند مراجعة الحوادث.
4. **سجّل الخروج من الأجهزة المشتركة أو غير الموثوقة** بعد كل جلسة.
5. **راجع جلساتك النشطة بانتظام** وألغِ أي جلسة لا تتعرف عليها.
6. **حافظ على وجود حسابين نشطين على الأقل بصلاحية super\_admin في جميع
   الأوقات.** لا تعطّل أحدهما حتى تتأكد أن الثاني يعمل بشكل صحيح.
7. **لا تمنح super\_admin للموظفين التشغيليين.** استخدم منح الصلاحيات
   الصريح لتحديد الوصول المطلوب بدقة.
8. **تعامل مع بيانات الاعتماد المؤقتة على أنها للاستخدام الواحد.** إذا
   أُرسلت بطريقة غير آمنة، عطّل الحساب وأنشئ حسابًا جديدًا.

## 11. إجراءات الطوارئ

### فقدان جهاز المصادقة مع توفر رموز الاسترداد

استخدم رمز استرداد في حقل TOTP عند تسجيل الدخول. بعد الوصول، انتقل فورًا
إلى **الحساب ← الأمان** لتوليد رموز استرداد جديدة وإعادة تسجيل TOTP.

### فقدان جهاز المصادقة دون رموز استرداد

تواصل مع مشغّل super\_admin الثاني. عليه:
1. إنشاء حساب جديد لك عبر **حسابات الموظفين ← إنشاء**.
2. تعطيل الحساب المحجوب.

إذا لم يكن هناك super\_admin ثانٍ متاح، الاسترداد يتطلب وصولًا مباشرًا
للسيرفر (SSH إلى الـ VPS):

```bash
cd /opt/wegosharm/resort-os
docker compose -f docker-compose.prod.yml exec backend \
  python -m app.admin_bootstrap
```

السكريبت يُنشئ حساب super\_admin جديد بكلمة مرور مؤقتة ورمز تسجيل.
استخدمهما لتسجيل الدخول وإتمام إعداد TOTP، ثم عطّل الحساب المحجوب من
الحساب الجديد.

### الاشتباه في اختراق حساب

1. **من حساب super\_admin الثاني فورًا:** ألغِ جميع جلسات الحساب المشتبه به
   (تبويب المستخدمون ← ابحث عن المستخدم ← تعطيل).
2. راجع سجل التدقيق للفترة المشتبه فيها.
3. أنشئ حسابًا جديدًا للشخص المتضرر فقط بعد فهم الحادثة.
4. إذا احتُمل تعرّض بيانات اعتماد النظام بأكمله للاختراق، غيّر `SECRET_KEY`
   في `backend/.env` على الـ VPS وأعد تشغيل خدمة الـ backend — هذا يُلغي
   جميع جلسات جميع المستخدمين في النظام فورًا:

```bash
# على الـ VPS — عدّل .env أولًا، ثم:
docker compose -f docker-compose.prod.yml restart backend celery_worker celery_beat
```
